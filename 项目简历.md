## 核心能力

### 一、记忆系统：用户画像 + 偏好持续积累

- 利用 Agent 框架的 **MemoryMiddleware + MemoryUpdater**，自动从对话中提取品牌偏好、历史选人记录
- 实现 **per-user 记忆隔离**：每位品牌方的类目偏好、预算范围、达人风格倾向持久存储
- 记忆数据通过 `<memory>` 标签**注入到每次模型请求的 System Prompt**，实现个性化推荐
- 支持记忆 staleness 检测 + **自动过期淘汰**，避免过时偏好影响推荐质量

### 二、上下文管理：Thread 级隔离 + 长对话压缩

- 每次选人任务绑定独立 **Thread**，享有独立的沙箱空间、对话历史和状态
- 利用 **SummarizationMiddleware** 对长对话自动压缩：保留近期消息窗口 + 将早期上下文压缩为 `summary_text`
- **ThreadState** 管理选人任务的完整生命周期：candidates → scores → feedback，跨轮次状态不丢失
- **DurableContextMiddleware** 确保子任务结果、技能上下文在摘要压缩后仍对模型可见

### 三、工程化：模块边界 + 依赖注入 + 测试覆盖

- **分层架构**：`deerflow.*`（框架层）与 `app.*`（业务层）单向依赖，Agent 工具通过**工厂模式 + 闭包注入**解耦
- **数据库**：5 张业务表，Alembic 迁移管理，SQLite/PostgreSQL 自适应
- **异常体系**：`DataPlatformError` / `MatchingError` / `ScoringError` 三层自定义异常 + Router 层统一捕获 + 降级响应
- **TDD 流程**：先写测试验证失败 → 实现 → 测试通过 → 提交，116 个测试覆盖全链路
- **配置管理**：`config.yaml` 驱动，`provider: mock | douyin` 一键切换数据源
- **前端工程**：Next.js App Router + Server Components 优先 + `pnpm check` 门禁

### 四、SubAgent 并行调度

- 定义 **4 个专业分析 Agent**：粉丝画像分析师 / 内容风格分析师 / 商业表现分析师 / 风险合规扫描师
- 利用框架的 **SubagentExecutor** 实现最多 3 个并行调度：Lead Agent 派发 → SubAgent 独立沙箱执行 → 结果汇聚
- **降级策略**：单维度超时标记"暂不可用"→ 用剩余维度算总分；全失败 → 纯数据对比

### 五、反馈闭环 + 评分自优化

- 品牌方提交合作反馈（星级 + 标签 + 文字）→ 对比**预测分 vs 实际评分**
- **梯度下降简化算法**：偏差 > 阈值时按学习率 0.05 微调四维权重并归一化
- **置信度递增**：`confidence` 随反馈量单调递增（≥0.6 才触发调整），权重逐渐收敛

## 面试话术

**Q: 项目中的记忆系统怎么设计的？**
> 框架的 MemoryMiddleware 会在每轮对话后自动触发 MemoryUpdater，用 LLM 从对话中提取品牌偏好、历史选人记录、预算范围等事实。记忆按用户隔离存储，通过 `<memory>` 标签注入到下次模型请求的 System Prompt 里。还有一个 staleness 机制——超过 180 天的旧事实会被标记为候选、经 LLM 审核后自动淘汰。

**Q: 上下文管理怎么处理长对话？**
> 每次选人任务是一个独立 Thread，有自己隔离的沙箱和状态。框架的 SummarizationMiddleware 在 token 接近上限时自动触发——保留最近的消息窗口，把早期上下文压缩成 `summary_text`。我利用了 DurableContextMiddleware 确保子任务结果和技能上下文在压缩后不丢失——它们被持久化到 ThreadState 然后重新注入到后续请求中。

**Q: Agent 工具怎么和业务层解耦的？**
> Agent 工具定义在框架层（`deerflow.tools`），但它们需要调用业务层的搜索和评分服务。我用了工厂模式——工具函数接收 adapter 和 engine 作为闭包参数，在 Gateway 启动时注入。框架层代码不引用任何 `app.*`，通过了边界检查。这和依赖注入是同一思路——上层组装，下层保持纯净。

**Q: 并行 SubAgent 怎么调度和容错？**
> 框架的 SubagentExecutor 原生支持最多 3 个并行任务。我定义了 4 种专业 SubAgent 类型注册到 Registry，Lead Agent 通过系统提示词中的工作流指引自动调度。容错方面我写了代码级 FallbackResult——单维度超时就标记"暂不可用"并用剩余维度算总分，全失败就降级为纯数据对比，保证用户不会看到空白页。

**Q: 反馈自优化的算法细节？**
> 品牌方评分 1-5 星，我转成百分制（rating×20）。FeedbackService 对比预测分和实际评分——偏差超过 10 分就触发调整。学习率 0.05，每次只微调权重并归一化到总和为 1。为了避免小样本噪声，confidence 低于 0.6 时只记录不调整。随反馈积累 confidence 递增，权重逐渐收敛。

## 数据流

```
品牌方输入需求
  → Memory 注入历史偏好
  → Agent 调用 search_influencers (Data Platform + Scoring Engine)
  → 派遣 3 个 SubAgent 并行分析 (Sandbox 隔离执行)
  → 任一完成 → 启动 Risk Scanner
  → 全部汇聚 → compare + recommend_report
  → 品牌方选定 + 合作反馈
  → FeedbackService 更新 influencer_scores + 微调权重
  → 下次搜索自动应用优化后的评分模型
```
