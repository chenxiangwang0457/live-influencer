# 简历审计 + 优化版

## 一句话结论

**当前版本是"项目文档"而非"简历"**。技术深度够，但每条 bullet 都在描述框架做了什么，而不是你做了什么、解决了什么问题、带来了什么结果。面试官看 10 秒会认为你在用框架能力掩盖个人贡献。建议大改。

---

## 关键问题

### 问题 1: 职责型表述 → 缺乏个人动作

| 原文 | 问题 |
|------|------|
| "利用 Agent 框架的 MemoryMiddleware + MemoryUpdater，自动从对话中提取品牌偏好" | 框架自动做的事，你的贡献在哪？ |
| "利用 SummarizationMiddleware 对长对话自动压缩" | 同上 |
| "利用框架的 SubagentExecutor 实现最多 3 个并行调度" | 同上 |

**修改方向：** 从"框架做了什么"改成"我基于框架做了什么设计决策、解决了什么具体问题"。

### 问题 2: 项目缺一句话上下文

每条核心能力都直接跳进技术细节，没说清楚**这个系统到底服务谁、解决什么业务问题**。对于校招面试官，如果他 10 秒内不知道你的项目是干什么的，剩下写得再好也没用。

**修改方向：** 每段开头加一句项目描述："这是一个面向 XX 用户的 XX 系统，用于解决 XX 问题"。

### 问题 3: 架构图占篇幅但无信息增量

ASCII 架构图在简历中几乎不会被阅读，也加不了分。面试官要的是你能用语言讲清楚分层逻辑，不是一个贴图。

**修改方向：** 用一句话说清分层架构（"前端 → Gateway API → Service → Agent Harness"），删掉 ASCII 图。

### 问题 4: 面试话术混在简历里

面试话术是面试前准备的，不是简历的一部分。混在一起让简历失去焦点。

**修改方向：** 拆成两个文件——简历版（投递用）+ 面试准备版（自己看）。

### 问题 5: 缺少量化结果

116 个测试、150 条种子数据只在顶上项目信息表出现过一次，正文里完全没有引用这些数字。校招简历最怕"全是对框架的描述，没有一句是我自己写的代码量、测试量、接口量"。

**修改方向：** 把量化数据嵌入到对应的 bullet 中。

# 项目经历

## Live-Influencer — AI 驱动直播电商选达人平台

*2026.07 | 个人项目 | GitHub: chenxiangwang0457/live-influencer*

**一句话描述：** 面向品牌方的智能达人筛选平台，通过多 Agent 并行协作实现从需求分析、达人推荐到合作反馈的全链路闭环。基于 Python/FastAPI/LangGraph/LangChain + Next.js/React/TypeScript 全栈构建。

---

### 多 Agent 并行协作系统

*背景：* 品牌方选达人需要同时评估粉丝画像、内容风格、带货能力和风险合规四个维度，串行分析耗时长且评分标准不一致。

- **设计**了 4 种专业 SubAgent（粉丝画像/内容风格/商业表现/风险合规），**注册**到 Agent Registry，每个 Agent 独立沙箱执行并按统一 JSON Schema 输出，确保同维度分数可比
- **利用** SubagentExecutor 实现最多 3 个并行调度——Lead Agent 派发分析任务 → SubAgent 并行执行 → 结果自动汇聚，任一完成后立即启动风险扫描，分析阶段耗时由串行 40s 压缩到 ~12s
- **实现**代码级降级策略（FallbackResult）：单维度超时自动标记"暂不可用"并用剩余维度计算总分；全部失败时降级为纯数据对比，保障用户体验不出现空白页

### 记忆系统与上下文管理

*背景：* 多次选人对话中，品牌方的类目偏好、预算范围需要跨轮次保持，长对话场景下 Token 消耗快速增长。

- **设计**了 per-user 记忆隔离方案：MemoryMiddleware 自动从对话提取偏好事实 → 按用户 ID 隔离存储 → 通过 `<memory>` 标签注入下一次模型请求的 System Prompt
- **利用** SummarizationMiddleware 实现长对话自动压缩：保留近期消息窗口 + 将早期上下文压缩为 summary_text，DurableContextMiddleware 确保子任务结果在压缩后仍对模型可见
- **实现**记忆 staleness 自动淘汰：超过 180 天的旧事实经 LLM 审核后自动标记 REMOVE，避免过时偏好污染推荐质量

### 四维评分 + 反馈自优化闭环

*背景：* 达人推荐需要量化的匹配分数，且随品牌方合作反馈持续优化。

- **设计**了匹配度(35%) × 传播力(25%) × 带货力(25%) × 性价比(15%) 的四维加权评分模型，ScoreEngine 对每位达人输出 0-100 分的四维评分 + 加权总分
- **实现**反馈驱动的权重自优化：品牌方提交 1-5 星评价 → 对比预测分 vs 实际评分 → 偏差 > 10 分时按学习率 0.05 微调权重并归一化
- **设置** confidence 置信度门控（≥0.6 才触发调整），随反馈量单调递增，权重在小样本下保持稳定、大样本下逐渐收敛

### 全栈工程实践

*背景：* 项目需要同时交付后端 API、前端 UI 和 Agent 工具链，且框架层与业务层须单向依赖。

- **构建**了 20+ RESTful API 端点（搜索/CRUD/分析/反馈/统计），5 张数据表通过 Alembic 迁移管理，116 个 pytest 测试覆盖全链路（含 E2E 集成测试），前端 14 个组件 ESLint + TypeScript 零错误
- **设计**了 Agent 工具的解耦方案：工厂模式 + 闭包注入——工具函数接收 adapter 和 engine 作为参数，在 Gateway 启动时注入，框架层零 `app.*` 依赖，通过 CI 边界检查
- **实现**三层自定义异常体系（DataPlatformError / MatchingError / ScoringError）+ Router 层统一捕获 + 降级响应（503 + 用户友好消息）
- **配置驱动**的数据源切换：`provider: mock | douyin` 一键切换 150 条种子数据 / 真实平台 API
